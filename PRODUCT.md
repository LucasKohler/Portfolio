# PRODUCT.md

## Product Summary

This project is a portfolio platform for **Lucas Kohler Marques**, a Software Engineer focused on fullstack systems, performance, architecture and AI-assisted development.

It must work as:

1. A professional presentation website.
2. A technical showcase on GitHub.
3. A foundation for adding real projects and case studies over time.

The project must be built as a real software product, not as a static one-page portfolio.

---

## Primary Goal

Convert visitors into professional opportunities.

Target opportunities:

- Software Engineer
- Fullstack Developer
- Frontend Developer
- .NET Developer
- React / Next.js Developer
- Backend .NET Developer
- AI-assisted development and automation-oriented engineering roles

---

## Secondary Goal

Show engineering maturity through the repository itself.

The codebase must demonstrate:

- Clean frontend architecture
- .NET backend architecture
- API design
- Dockerized development
- Dynamic routing
- Extensible project data model
- Reusable components
- Clear documentation
- Separation of concerns
- Ability to evolve into a larger product

---

## Product Positioning

**Software Engineer focused on fullstack systems, performance and AI-assisted development.**

This positioning should be visible in:

- Hero copy

- Technical stack

## User Journey

The intended journey:

```txt
Hero
  ↓
Engineering value proposition
  ↓
Featured Engineering Work
  ↓
Projects Index
  ↓
Project Detail
  ↓
Contact CTA
```

The visitor should see evidence before being asked to contact.

---

## Product Requirements

### Home

Must answer quickly:

- Who is Lucas?
- What does he build?
- Which stack does he use?
- Why is he credible?
- Where can I see his work?

### Projects Index

Must work as a scalable catalog of:

- Real projects
- Case studies
- Technical experiments
- AI workflow experiments
- Architecture notes

It must not feel empty or fake.

### Project Detail

Must document each project honestly.

It should show:

- Purpose
- Overview
- Technical highlights
- Implementation notes
- Current status
- Next steps
- Stack

It should not fake results.

### Backend API

The backend exists to show real .NET capability and to centralize project data.

It should expose:

- Projects
- Project details
- Contact endpoint
- Health endpoint
- API documentation

### Docker

Docker exists to show deploy-ready thinking and environment consistency.

The project must be runnable locally with Docker Compose.

---

## Contact Strategy

The Contact section is the main conversion point.

The navbar `Contact` item should scroll to the Contact section.

The `Contact Me` button should conceptually use a `mailto:` link or a future backend contact flow.

GitHub, LinkedIn, Email and Resume should also exist in the footer.

---

## Future Evolution

The architecture should allow future additions:

- New projects
- Project tags
- More detailed case studies
- Blog/technical notes
- Bilingual content
- Admin/content management
- Database-backed project data
- Contact form provider
- Analytics
- CI/CD
- Production deployment

Do not overbuild these in version 1. Prepare the architecture so they can be added later.
